What's this

ApolloGeese represents a design philosophy, a collection of essential web-application back-end modules and one sexy language.

The current proof-of-concept implementation has been done in Mono C#. This means that you can use this neato fresh software stack on pretty much any platform. Without recompiling.

Design philosophy

In any given software design, more than 90% of the program flow consists of merely a handful of similar elements chained together in a hierarchical fashion, exchanging typically trivial amounts of information.

We can use this knowledge to develop a simpler, quicker way of creating software than we're used to from conventional development platforms such as Python, PHP and ASP.Net. This development-time speedboost could be achieved while imposing only some minor, creative limitations on developers.

We'd have to decide upon a protocol outlining the behavior and structure of non-distinct elements in software, and a protocol outlining the communication between those elements.

For ApolloGeese, precisely this has been done; non-distinct elements will be referred to as Services, communication between Services is done by means of Interactions

Services

Services expose:

  • A Description of the service, wherein they aptly describe their purpose in relation to their current situation within a program flow.
  • A single entry point used to provide the service with context by means of an Interaction originating from hierarchically upstream services.
  • Multiple named exit points used to propagate richer context by means of (new) Interactions to hierarchically downstream services.

Interactions

Interactions expose:

  • Means to store information on a key-value basis, including nested key-value stores. (Though admittedly, don't you dare to even consider it)
  • Means to access ancestral Interactions; Interactions that have triggered the creation of the current one.
  • Means to access information stored on both the current and ancestral Interactions.

Modules

Well over twenty modules and have already been developed in accordance to this philosophy.

Below, six modules most relevant to web development have been highlighted:

HTTP Server

Listens on one or multiple ports and prefixes for incoming HTTP requests and creates a context wherein an URL-queue, header-data and body-data exist.

SiteSubsection

Uses the URL-queue from existing context to deviate into an appropriate branch of the program flow. Enriches the context with the current URL-section name.

Template

Uses either existing context or underlying branches to fill in template gaps.

GetFieldReader

Parses the query-section of the url (after the question mark), verifies this input and branches onwards depending whether or not verification was successful. It enriches the context with the parsed and sanitized input.

PostFieldReader

Shares many traits with the GetFieldReader, but instead parses and verifies the HTML POST-body into context.

MySqlQuerier

Uses values from context to fill in a parameterized query. It branches for each result, enriching context with acquired row values.

Tying it together

We now have a philosophy, and a handful of very useful modules, but no means to tie them together other than Mono-languages. And let's be fair: the temptation to deviate from the design philosophy would soon be too great with such a complex language.

Furthermore, the Mono/.NET languages would serve a problem too so soon as (I failed to mention this earlier) the actual implementation is done in C.

So I proudly present you:

CLON - The language

Because the correct answer to the question "Should I make my product sound like a spaceship or an STD?" can and should only be "Both".

CLON is short for 'C-Like Object Notation'. That's because it looks a little like C, but it notates objects. It can define a flow of operation without conditionals, but shares syntactical traits with JSON otherwise. That means it's hierarchical and blows curly-ass brackets all over the place.

I will now explicitly restate that CLON is merely an 'Object Notation'. It is not a programming language. If you dare to claim otherwise, I dare you to make a turing machine in it. (Instructing ApolloGeese to behave like a Turing machine doesn't count, ew.)
